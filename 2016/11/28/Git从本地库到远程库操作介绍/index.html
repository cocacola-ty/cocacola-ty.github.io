到这里本地版本库的基本使用就已经完成了。

3 使用分支

现在所有的操作都是在master分支上面进行的。来使用一下分支功能。

情景：现在有功能正在开发，突然又有新功能需要开发。这样可以开一条分支来进行新功能的开发。这样保证不干扰现在的一半的开发。

3.1 先创建一条分支

这样就创建了一条名为branch1的分支。不过这只是创建了一条分支，并不会自动切换到该分支上面。新功能的开发就在这条分支上面。
这条创建的分支branch1是根据当前所在的分支(master)创建的。

这是因为创建分支是基于当前所在分支的最新提交作为新分支的开始，也就是说当前分支的最新提交的状态就是新分支的起点。

现在切换回master分支看一下情况

会发现此时工作目录中只有file1.txt

因为当切换分支后，工作目录会同时更改为该分支状态的目录。此时master分支中并没有branch1.txt文件，所以工作目录中只有master分支上面的file1.txt文件。

分支上的操作是不会互相干扰的。branch1.txt文件是在branch1分支上面添加的，所以master分支上面是没有的。这样两条分支各自有一种工作状态。

4 合并

上面创建了一条分支branch1用于开发一个单独的功能，此时该功能已经开发完成。master分支上面的功能也开发完成。由于分支之间是相互独立的，所以branch1分支上面的代码没有master上面开发的功能。同理master上面也没有branch1的功能。

下面就通过合并来将这两个分支上的功能集成起来。我们把branch1的代码合并到master上来。

远端仓库的配合使用

此时需要有一个远端仓库，如github等。
此处的远端库使用https://github.com/cocacola-ty/TestRep.git

克隆远端仓库

获取远端仓库的内容我们需要把远端仓库克隆到本地。

克隆下来之后本地会有一个和远端仓库内容一样的仓库。

这里的一样是指仓库中的工程内容是一样的，一些版本库的私有配置如钩子等是不一样的。

Note

  • 远程仓库中的分支到克隆下来的本地版本库中成为远程跟踪分支。如远程的库中的master分支，在本地库中就成为了origin/master。这两条分支是关联的。

  • 当克隆下来之后，本地库中会自动创建一条master分支并切换到该分支，该分支与远程库中的master的内容是一样的。

  • 在本地所做的修改默认都是在本地的master分支上面的修改。
